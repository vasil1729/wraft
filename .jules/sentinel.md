## 2026-04-12 - Prevent User Enumeration via Timing Attacks in Authentication Flows
**Vulnerability:** The application was vulnerable to timing-based user enumeration attacks. In `SessionController` and `UserController`, when an invalid email was provided, the code would exit early without executing the computationally expensive `Bcrypt.verify_pass()`. This timing difference allowed an attacker to enumerate which email addresses exist in the system. Additionally, `SessionController` would reveal if a user account was deactivated even if the provided password was incorrect, which is a logical leak. Finally, `FallbackController` leaked different error messages for non-existent vs existing users with incorrect passwords.
**Learning:** Short-circuiting execution paths (e.g., using `with` statements) upon failed user lookups can inadvertently leak information through timing discrepancies. It is crucial to ensure that time-consuming operations, like password verification, occur in all paths (success and failure). Furthermore, business logic checks (like `is_deactivated`) must only occur *after* successful authentication to prevent logical information leaks. Generic error messages must always be used across the board.
**Prevention:**
1. Use `Bcrypt.no_user_verify()` to simulate the timing of a password hash verification when a user lookup returns `nil` or `{:error, :invalid_email}`.
2. Refactor `with` statements or use nested `if`/`case` constructs in authentication flows to guarantee exactly one slow crypto operation runs.
3. Check business rules (like account active status) only after `Bcrypt.verify_pass()` returns `true`.
4. Ensure error handlers (like `FallbackController`) map different authentication failures to the exact same generic error message.
