## 2024-05-24 - [Timing Attack in Authentication]
**Vulnerability:** Timing attack due to failed user lookup in authentication flow. When looking up a user during sign-in, if the user does not exist, the request returns faster because the expensive password hashing function is skipped, allowing an attacker to enumerate valid email addresses.
**Learning:** In Elixir authentication flows, short-circuiting `with` statements upon failed user lookups can cause timing leaks. `Bcrypt.no_user_verify()` must be explicitly called on these specific failure paths (e.g., `nil` or `{:error, :invalid_email}`) to simulate password hashing delays. Additionally, verify the user's password (`Bcrypt.verify_pass`) before checking business logic status flags like `is_deactivated`. Placing status checks before password verification introduces a timing attack vector that allows enumeration of account statuses.
**Prevention:** To prevent user enumeration via timing attacks, authentication logic must use `Bcrypt.no_user_verify()` when a user lookup fails to simulate password hashing time. Also ensure that status checks are performed after verifying passwords to prevent enumeration of account statuses.
