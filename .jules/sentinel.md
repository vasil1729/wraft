## 2024-05-24 - [User Enumeration Timing Attack Prevention]
**Vulnerability:** User enumeration is possible via timing attacks during login in both `WraftDocWeb.SessionController` and `WraftDocWeb.Api.V1.UserController`. If an invalid email is provided or an account is deactivated, the authentication flow fails fast, bypassing the password hashing step (`Bcrypt.verify_pass`). This results in a noticeably faster response, allowing an attacker to determine if an email exists in the system or is deactivated.
**Learning:** In Elixir/Phoenix authentication flows using `bcrypt_elixir` (or similar libraries), it's crucial to explicitly simulate the password hashing delay when a user lookup fails or when an account check (like `is_deactivated`) fails before password verification. The `with` statement's early exit behavior inherently creates this timing leak if not handled carefully.
**Prevention:** Always call `Bcrypt.no_user_verify()` on the failure path of authentication controllers when a user is not found or when validation fails prior to password checking. Additionally, ensure that authentication endpoints return standardized, generic error messages (e.g., `{:error, :invalid}`) instead of specific ones (e.g., `{:error, :invalid_email}`) to prevent enumeration via error payloads.
