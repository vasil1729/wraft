## 2024-03-30 - Prevent Timing Attacks in Authentication Flows
**Vulnerability:** Timing attacks and user enumeration vulnerabilities in authentication controllers (`WraftDocWeb.SessionController` and `WraftDocWeb.Api.V1.UserController`). If an email lookup fails, the flow short-circuited quickly, which allowed an attacker to determine if an email existed in the system based on response time. Also, checking business logic flags (like `is_deactivated`) before verifying the password introduces a similar vulnerability.
**Learning:** During Elixir authentication flows, we must ensure password verification is performed exactly once, even if the user lookup fails. The `with` statement must include an `else` block to call `Bcrypt.no_user_verify()` on failure paths (e.g. `nil` or `{:error, :invalid_email}`). We also must return standard error messages like `{:error, :invalid}` instead of specific errors like `{:error, :invalid_email}` to prevent leaking whether an email exists. Also, status checks (like `is_deactivated`) must come *after* password verification.
**Prevention:** Always add an `else` block to `with` statements handling authentication. In the `else` block, explicitly match the missing user case and invoke `Bcrypt.no_user_verify()`, then return a generic error. In controllers handling admin sessions, ensure any `is_deactivated` flag check occurs after `Bcrypt.verify_pass` succeeds.
