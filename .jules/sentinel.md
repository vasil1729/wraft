## 2024-03-24 - [User Enumeration via Timing Attacks in Elixir Auth Flows]
**Vulnerability:** User enumeration via timing attacks in Elixir authentication flows due to short-circuiting `with` statements and conditionally skipping `Bcrypt.verify_pass()`. If a user lookup failed (`nil` or `{:error, :invalid_email}`), the `with` statement would early-exit without simulating the delay of a password hash. Furthermore, status checks like `is_deactivated` were happening *before* password verification, which also introduced timing discrepancies. In addition, the `generate_token` password reset flow returned specific errors if the email did not exist.
**Learning:** In Elixir authentication flows, short-circuiting logic (e.g., `with` or `cond`) upon failed user lookups can cause severe timing leaks, allowing attackers to guess valid usernames/emails. We also learned that placing business logic status checks *before* password verification introduces another timing attack vector that allows enumeration of account statuses. Finally, doing `Bcrypt.no_user_verify()` on the unhappy path of a password reset flow (e.g., invalid email) introduces a severe CPU-exhaustion Denial of Service (DoS) vulnerability.
**Prevention:** Explicitly call `Bcrypt.no_user_verify()` on specific failure paths (e.g., `nil` or `{:error, :invalid_email}`) to simulate password hashing delays. Do not call it if password verification fails, to avoid double-hashing vulnerabilities. Prefer nested `if` statements over `with` or `cond` structures in auth flows to guarantee computationally expensive operations like `Bcrypt.verify_pass()` are evaluated exactly once. Always return a generic 'Success' message with a 200 OK status on `generate_token`, even if the user email lookup fails. Finally, verify the user's password (`Bcrypt.verify_pass`) *before* checking business logic status flags like `is_deactivated`.
