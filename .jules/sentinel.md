## 2024-04-03 - Timing and Enumeration Vulnerability in Authentication Controllers
**Vulnerability:** In `SessionController`, the `is_deactivated` flag was checked prior to password verification, which allowed an attacker to determine if an email belongs to an existing, deactivated account using an arbitrary password. In addition, when the email didn't exist (`nil` user), there was no hashing delay, opening up a timing attack vector to enumerate existing valid emails.
**Learning:** Business logic status checks (like `is_deactivated`) must be performed *after* successful password verification. Additionally, authentication logic needs to use `Bcrypt.no_user_verify()` explicitly when a user lookup returns `nil` to simulate password hashing delay, preventing timing attacks.
**Prevention:** Always verify credentials before checking account state. In Elixir `with` statements handling authentication, ensure `Bcrypt.no_user_verify()` is triggered in the `else` block specifically for `nil` or `{:error, :invalid_email}` results from user lookup.
