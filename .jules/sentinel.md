## 2024-05-20 - [Fix timing leak in authentication controllers]
**Vulnerability:** User enumeration timing attack. `with` statements in authentication controllers (`SessionController`, `UserController`) short-circuited upon failed user lookups, skipping the computationally expensive `Bcrypt.verify_pass()`. This allowed attackers to distinguish between valid and invalid emails by observing response times.
**Learning:** In Elixir, returning early from `with` statements when a user is not found bypasses password hashing, causing timing differences. `Bcrypt.no_user_verify()` must be explicitly called on the `nil` or `{:error, :not_found}` / `{:error, :invalid_email}` paths at the controller level to simulate the hashing delay. Do NOT call it on failure paths where password verification has already failed (to avoid double hashing) or in password reset logic (to avoid DoS).
**Prevention:** Always add an `else` clause to `with` statements in authentication endpoints that explicitly catches user-not-found errors and calls `Bcrypt.no_user_verify()` before returning the generic error response.
