## 2026-03-25 - Prevent Timing Attacks in Elixir with Statements and Enumeration in Password Reset
**Vulnerability:** Timing attacks due to early exits in `with` statements and `cond` logic (e.g., checking `is_deactivated` before `verify_pass`) allows user enumeration. Password reset returning conditional responses allows email enumeration.
**Learning:** In Elixir, using `Bcrypt.no_user_verify()` is crucial on failure paths (`nil` user or `{:error, :invalid_email}`) to simulate password hashing delay. However, it should NOT be used on unhappy paths of password resets, which naturally don't perform hashes. Also, `with` statements must explicitly handle the `{:error, :invalid_email}` case to inject this delay.
**Prevention:** Always verify passwords before checking business logic status flags. Use `Bcrypt.no_user_verify()` explicitly on missing-user paths. Always return generic "Success" messages with 200 OK statuses for password reset requests, regardless of whether the email exists.
## 2026-03-25 - Prevent WithClauseError crashes and timing vulnerabilities
**Vulnerability:** Adding a single `else` match in an Elixir `with` statement removes the automatic catch-all functionality, leading to `WithClauseError` (HTTP 500 crashes) on unhandled error tuples. Double execution of Bcrypt hashes when fixing timing attacks leads to even worse enumeration bugs.
**Learning:** Always include a catch-all `error -> error` (or similar) when modifying the `else` block of a `with` statement. When mitigating timing attacks, trace the logical path to ensure the slow operation (e.g., `Bcrypt.verify_pass()`) is invoked exactly once for both valid and invalid user paths.
**Prevention:** Avoid `cond` logic for authentication. Prefer single variable assignment and conditional branching (`if/else`) where the slow hash verification is explicitly controlled, or `with` statements where the error matching is exhaustive.
