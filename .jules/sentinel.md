## 2024-05-24 - [Timing Attack in Elixir Authentication Flows]
**Vulnerability:** The application used `with` statements in authentication controllers (`SessionController` and `UserController`) that short-circuited when a user could not be found by email. This skipped the `Bcrypt.verify_pass/2` computation, resulting in a significantly faster response time for non-existent emails compared to existing ones, allowing attackers to enumerate valid email addresses via timing attacks.
**Learning:** In Elixir, `with` statements are elegant but dangerous for authentication because early exits on failure paths bypass computationally expensive operations like password hashing. Using `Bcrypt.no_user_verify()` is necessary on the unhappy path (e.g., user not found) to simulate the hashing delay, and password verification must be checked *before* other business logic flags like `is_deactivated`.
**Prevention:** Always refactor authentication flows to use explicit branching (like nested `if` or `case` statements) instead of `with`. Ensure `Bcrypt.no_user_verify()` is called precisely when user lookup fails, and `Bcrypt.verify_pass()` is executed unconditionally for existing users before any status checks. Avoid calling `Bcrypt.no_user_verify()` if password validation fails, to prevent double-hashing Denial of Service.
