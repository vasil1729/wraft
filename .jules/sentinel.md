## 2025-02-15 - [Prevent User Enumeration via Timing Attacks in Authentication]
**Vulnerability:** User enumeration via timing attacks during the authentication flow. When looking up a user, if the user was not found, the operation returned immediately (`nil` or `{:error, :invalid_email}`). This allowed an attacker to enumerate valid email addresses because requests for invalid emails returned significantly faster than valid ones (which performed an expensive password hash check). Additionally, `{:error, :invalid_email}` could leak the existence of the user via the `WraftDocWeb.FallbackController`.
**Learning:** Returning early on failed user lookups in a `with` statement creates a timing disparity. To prevent this, when a user is not found, we must explicitly call `Bcrypt.no_user_verify()` to simulate the time delay of the expensive password hashing step before returning a generic error (e.g., `{:error, :invalid}`).
**Prevention:** Always implement `Bcrypt.no_user_verify()` in the error handling of the controller for authentication flows when user lookup fails. Ensure error messages returned to the user are generic and do not distinguish between a missing account and an incorrect password.
