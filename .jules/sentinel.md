## 2024-03-24 - User Enumeration and DoS Vectors in Authentication Flows
**Vulnerability:** The login and password reset flows allowed attackers to enumerate valid email addresses. The login flow leaked user existence via a timing attack (lacking a simulated password hashing delay when a user email was not found). The password reset endpoint explicitly leaked user existence by returning distinct responses (`404` vs `200`). Furthermore, attempting to fix the timing attack on the password reset endpoint by adding `Bcrypt.no_user_verify()` created a CPU-exhaustion DoS vector, because the happy path for a password reset *does not* verify a password, so forcing a bcrypt hash on the unhappy path reverses the timing attack and introduces massive overhead.
**Learning:** Elixir `with` statements naturally short-circuit on the first non-matching pattern. In authentication flows where a password *is* verified (like login), short-circuiting a user lookup before the password verification step introduces a timing leak. However, in flows where passwords are *not* verified (like password reset requests), adding artificial delays like `Bcrypt.no_user_verify()` introduces dangerous DoS vulnerabilities and exacerbates timing differences.
**Prevention:** Always include an `else` block in authentication `with` statements (e.g., `login`) to explicitly catch failed user lookups (e.g., `nil` or `{:error, :invalid_email}`) and call a dummy verification function like `Bcrypt.no_user_verify()`. Ensure you match exactly what the lookup returns to avoid `WithClauseError` crashes. For endpoints that *do not* verify passwords (like reset requests), never add artificial hashing delays; instead, ensure they return a generic 200 OK success message rapidly, regardless of whether the email exists.
