## 2024-05-27 - [Authentication Timing Attacks and DoS in Elixir]
**Vulnerability:** User enumeration is possible via timing attacks during authentication because `is_deactivated` checks were occurring *before* password verification, and invalid emails skipped password hashing. Furthermore, forcing Bcrypt hashing on invalid emails during password reset flows (`generate_token`) introduced a CPU-exhaustion DoS vulnerability.
**Learning:** In Elixir authentication flows, short-circuiting `with` statements upon failed user lookups can cause timing leaks. For login, explicitly call `Bcrypt.no_user_verify()` on failure paths (e.g., `nil` or `{:error, :invalid_email}`) to simulate password hashing delays. However, DO NOT use `Bcrypt.no_user_verify()` on the unhappy path of a password reset flow because the happy path only generates a token without verifying a password.
**Prevention:** Always verify the user's password (`Bcrypt.verify_pass`) before checking business logic status flags like `is_deactivated`. Ensure authentication endpoints always return standardized generic error messages (e.g., `{:error, :invalid}`).
