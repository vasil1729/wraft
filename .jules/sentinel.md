## 2024-05-24 - [CRITICAL] Command Injection via Gnuplot Script Evaluation
**Vulnerability:** In `lib/wraft_doc/documents/documents.ex`, `generate_gnu_gantt_chart/2` generates a gnuplot script by performing simple string replacements for user-controlled inputs like `title` and `dest_path`. The script is then executed by the system shell via `System.cmd("gnuplot", ...)` which does not prevent command injection since gnuplot evaluates the script contents as gnuplot commands, and gnuplot allows executing shell commands (e.g., via `system()`).
**Learning:** Even when using `System.cmd/2` with a list of arguments, replacing raw user input directly into a script file (`.plt` for gnuplot) which acts as an interpreter can lead to command injection if the variables are not properly escaped. The system considers the input as commands. In addition, the generated script is placed in a static `/temp/` directory leading to race conditions between requests. Also, the output directory paths are shared. The memory guidance specifically mentions this issue and the need for escaping variables in `.plt` files and creating UUID-based subdirectories for temporary files. We must also use `Path.basename/1` for the uploaded filename.
**Prevention:**
1. Validate and sanitize user-provided values (like the title and upload filename). Sanitize `title` by strictly restricting its characters. Use `Path.basename/1` to prevent directory traversal for `filename`.
2. Escape all string content before interpolating into the `.plt` script to prevent gnuplot script injection.
3. Use a unique temporary directory using `Ecto.UUID.generate()` for both `temp/gantt_chart_input/` and `temp/gantt_script.plt` to avoid race conditions. Ensure the directories are created safely. Use `try/after` blocks to properly clean up temporary script and input files to avoid disk leaks.
4. Note that `generate_gnu_gantt_chart` caller functions expect `{output_string, exit_status}` from `System.cmd`.
