## 2024-05-24 - [User Enumeration and Timing Attack Fixes]
**Vulnerability:** The application was vulnerable to user enumeration via timing attacks during login and via direct responses in password reset flows. Specifically, `SessionController` and `UserController` failed to simulate password hashing time (`Bcrypt.no_user_verify()`) when a user lookup failed, and checked for deactivated accounts before hashing passwords. `UserController.generate_token` explicitly returned an error if an email didn't exist instead of a generic success message.
**Learning:** In Elixir/Phoenix auth flows, you must execute `Bcrypt.no_user_verify()` explicitly in the controller on paths where user lookups fail (`nil` or `{:error, :invalid_email}`) to ensure the response time is indistinguishable from valid user attempts. Additionally, always verify passwords *before* checking business logic flags (like `is_deactivated`) to prevent attackers from determining account statuses based on fast returns.
**Prevention:** Always use a `with` statement that first finds the user, then verifies the password, and only *then* evaluates account flags. In the `else` block, explicitly handle missing users with `Bcrypt.no_user_verify()` and generic error messages. For password resets, always return generic success responses regardless of whether the email exists.
