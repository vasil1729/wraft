## 2024-04-25 - Timing attacks in authentication flows due to `with` statements and early exits
**Vulnerability:** The application was vulnerable to timing attacks and user enumeration because authentication flows exited early on invalid emails without simulating password hashing time, and evaluated business logic (`is_deactivated`) prior to verifying the user's password. The use of Elixir `with` statements exacerbated this by causing early returns before expensive operations (`Bcrypt.verify_pass()`) were evaluated. Additionally, the fallback controller translated generic errors to specific email errors, leaking valid vs invalid email addresses.
**Learning:** In authentication flows, expensive operations (like password hashing) must take a consistent amount of time regardless of whether a user lookup succeeds or fails, and regardless of their business status. Relying on `with` statements can be dangerous if the "unhappy path" bypasses this cost, creating a timing leak. Furthermore, business flags (e.g. `is_deactivated`) should only be checked *after* the password is confirmed correct to prevent status enumeration.
**Prevention:** Refactor `with` statements in authentication logic to use nested `if` or `case` statements. Ensure `Bcrypt.no_user_verify()` is explicitly called when a user lookup fails, and always evaluate `Bcrypt.verify_pass()` before any status checks. Always return generic `{:error, :invalid}` tuples for invalid logins.
