## 2024-04-16 - Prevent User Enumeration via Timing Attacks in Authentication
**Vulnerability:** The authentication flows in `SessionController` and `UserController` were vulnerable to user enumeration via timing attacks. The code used `with` statements that short-circuited when a user was not found, skipping the computationally expensive `Bcrypt.verify_pass/2` function. This caused failed logins for non-existent users to return much faster than failed logins for existing users, allowing attackers to guess valid email addresses.
**Learning:** In Elixir, using `with` or `cond` statements in authentication paths can inadvertently introduce timing leaks if the heavy operation (password hashing) is skipped when earlier lookups fail. Additionally, returning specific error messages like `{:error, :invalid_email}` from fallback controllers leaks account existence.
**Prevention:** Always use `Bcrypt.no_user_verify()` on failure paths where user lookup fails to simulate password hashing delays. Use nested `if` or `case` statements to guarantee that either `verify_pass` or `no_user_verify` is evaluated exactly once. Always return standardized, generic error messages (e.g., `{:error, :invalid}`) for all authentication failures.
